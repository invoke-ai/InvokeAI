# Webv2 module architecture

Webv2 uses four top-level module groups:

- `app/` owns providers, routing, boot, and composition roots.
- `platform/` owns application infrastructure with no workbench or feature dependency.
- `workbench/` owns the atomic project aggregate, layout, shell, widget runtime, and extension runtime.
- `features/<domain>/` owns domain behavior behind a small set of public top-level entry modules (`features/<domain>/index.ts` first among them).

Dependency direction is `app -> workbench/features -> platform`. Feature callers import only a feature's public entry modules — the top-level modules each feature registers in `src/architecture/featureInterfaces.ts` (`index` is implicitly public; an unregistered feature or module is private, fail-closed); everything under `core/`, `data/`, `ui/`, and runtime implementation is private. A feature may group substantial behavior under `core/`, `data/`, and `ui/`; pure cores do not import React, HTTP, or UI code. Production canvas callers use `canvas-engine/api.ts`, not the engine implementation root.

## App and Platform ownership

Ownership is based on invariants, not the first caller:

| Module                 | Owned implementation                                                                                                                                  | Must not own                                                    |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------- |
| `app/`                 | `main`, provider assembly, routes and guards, route runtimes, adapter selection, cross-module dependency construction, theme/language DOM controllers | Domain behavior, transport implementation, reusable state or UI |
| `platform/transport`   | HTTP primitives and URL resolution, injected auth hook, Socket.IO mechanics, connection snapshots                                                     | Auth token policy, Queue/model/node event interpretation        |
| `platform/state`       | React-free listener/store/selector cores and separate React hook adapters                                                                             | Domain stores or aggregate state                                |
| `platform/query`       | Query client construction and defaults                                                                                                                | Feature query keys, DTOs, invalidation policy                   |
| `platform/i18n`        | i18next bootstrap and supported-language metadata                                                                                                     | Persisted preference ownership                                  |
| `platform/ui`          | Chakra system/theme, reusable controls, toaster, brand mark                                                                                           | Workbench widgets or feature-specific views                     |
| `platform/runtime`     | Domain-neutral build/runtime metadata                                                                                                                 | Route or feature policy                                         |
| `workbench/backend`    | Queue/node/model progress interpretation and transient domain stores pending their feature migrations                                                 | HTTP, sockets, connection mechanics, generic stores             |
| `workbench/components` | Workbench hydration and Queue progress presentation                                                                                                   | Reusable control or theme implementation                        |

The root of `src/` contains declaration files only. Runtime boot files live in App, the old root entry paths do not exist, and `index.html` names `src/app/main.tsx` directly. `@theme` resolves to Platform UI theme implementation; all other Platform callers use `@platform/*`. No compatibility path exists for moved App, HTTP, socket, external-store, selector, theme, i18n, or reusable-UI modules.

Platform transport receives authentication through `app/configureDependencies.ts`: Auth owns token storage and unauthorized-session behavior, while Platform only requests a token and reports a 401. The Platform socket publishes raw events and connection transitions; Queue and other domains interpret those events. App constructs the production Queue runtime from Queue, Workbench, Gallery, and Workflow adapters.

`pnpm architecture:check` is the executable source of truth for these dependency directions. It uses the TypeScript parser to classify static imports, type imports, dynamic imports, re-exports, and inline imported types after resolving aliases and relative paths. Temporary violations must be recorded in `src/architecture/migrationExceptions.ts` with an owner and a removal ticket; architecture completion requires that registry to be empty.

The queue pilot is the first migrated feature. Its private `QueueBackendPort` is the adapter seam; public callers use domain reads, commands, and production runtime factories from `@features/queue`. Private snake_case HTTP DTOs map explicitly to camelCase Queue read models before crossing that interface. TanStack Query owns the global backend queue read model, and non-React runtimes own reconciliation and realtime orchestration. Persisted per-project queue history remains in the Workbench aggregate.

Workbench persistence follows the same lifetime rule. `createWorkbenchPersistenceRuntime` owns hydration ordering, debounced autosave, revision/generation checks, conflict application, reconnect replay, status publication, and disposal without importing React. `createSyncedWorkbenchPersistence` constructs the local/remote project synchronization state for one mounted Workbench; no process-wide maps, sets, flags, or caches hold that state. `WorkbenchProvider` only constructs these lifetimes, starts/disposes the runtime through the mount lifecycle adapter, and selects aggregate hydration state.

## Contract ownership

Contracts live with the module that owns their invariants. The retired `workbench/types.ts` path is an architecture violation; Workbench aggregate, widget, Canvas, persistence, settings, and feature contracts each have an owner-local canonical path. Persisted schemas and backend DTOs remain private versioned/wire values and cross seams only through explicit serializers, hydrators, or mappers.

Feature callers import a feature's public entry modules; imports of another feature's `core/`, `data/`, `ui/`, or runtime implementation are rejected outside a ticketed migration exception. Wildcard exports may aggregate files inside one owner, but may not cross an ownership seam. The contract caller matrix and development-invalidation budgets are generated by `pnpm architecture:check` and `pnpm test:performance:build` respectively.

Compatibility exports under `workbench/` are temporary. New behavior belongs behind the target module interface and compatibility files must not acquire new implementation logic.

## Workbench ownership destinations

`src/architecture/workbenchOwnershipManifest.json` is the executable current-to-target ownership manifest. `pnpm architecture:check` proves that every production Workbench module has exactly one target owner and writes the detailed import/export inventory to `artifacts/architecture/workbench-ownership-inventory.json`. The generated inventory is regenerated as modules change; tests and type-test fixtures are intentionally excluded.

| Current Workbench path                                                                               | Target owner         | Split or locality decision                                                                                                                                |
| ---------------------------------------------------------------------------------------------------- | -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| aggregate root modules, `projects`, `persistence`, `settings`                                        | Workbench            | Atomic project/account state and its persistence policy remain local to the aggregate. Persistence mechanics move behind the aggregate's adapter.         |
| `shell`, `components/WorkbenchSplashScreen`, `launchpad` shell, `widget-frame`, non-domain `widgets` | Workbench            | Route-independent shell, layout, widget hosts, and registration are Workbench implementation.                                                             |
| `extensions`, `hotkeys`, `diagnostics`                                                               | Workbench            | These enforce the Workbench extension runtime; they are not independent vertical domains.                                                                 |
| `canvas-engine`, `canvas-operations`, Canvas/layers widgets, root Canvas modules                     | Workbench            | Canvas is a deep internal Workbench module behind `canvas-engine/api.ts`, not another top-level Feature.                                                  |
| `image-actions`, `images`                                                                            | Workbench            | Cross-surface image orchestration and transient display stay internal until a second independent implementation justifies a seam.                         |
| `auth`, `users`, Launchpad Users page                                                                | `feature:identity`   | Session/token policy and user administration form one identity vocabulary.                                                                                |
| `models`, Launchpad model manager/page, `backend/modelLoadStore`                                     | `feature:models`     | Model catalog, install/load lifecycle, and model administration co-locate.                                                                                |
| `customNodes`, Launchpad node manager/page, `backend/nodeExecutionStore`                             | `feature:nodes`      | Node catalog/template and execution lifecycle co-locate.                                                                                                  |
| `backend` except model/node stores, queue root modules, queue widget, Queue progress component       | `feature:queue`      | Queue owns server-event interpretation, transient progress, submission, reconciliation, and Queue UI. Persisted project queue history stays in Workbench. |
| `gallery`, Gallery widget                                                                            | `feature:gallery`    | Gallery data, selection policy, and Gallery UI form one vertical module.                                                                                  |
| `generation`, Generate widget                                                                        | `feature:generation` | Generation settings, graph compilation, and Generate UI form one vertical module.                                                                         |
| `workflows`, `graph-preview`, Workflow widget                                                        | `feature:workflow`   | Workflow documents, compilation, preview, editor, and library form one vertical module.                                                                   |
| `upscale`, Upscale widget                                                                            | `feature:upscale`    | Upscale settings, validation, compilation, and UI form one vertical module.                                                                               |
| `hooks/useMountEffect`, root `useModifierHeld`                                                       | Platform             | These are domain-neutral React lifecycle/input adapters and must have no Workbench import.                                                                |

The mixed-folder rules are longest-prefix rules in the manifest, so the Backend, Components, Launchpad, and Widgets splits are explicit rather than inferred from imports. Root modules are listed individually. Compatibility re-exports keep the target owner of the implementation they forward and are deletion work, never a fifth owner.

The target dependency graph intentionally records the current migration cycle: Workbench and the eight destination Features are one transitional strongly connected component. That is evidence for the cycle-breaking sequence, not permission for the destination architecture to retain the cycle. The migration must replace implementation imports with public module interfaces or composition-root adapters until the only direction is `app -> workbench/features -> platform`.

Feature interfaces remain ordinary module interfaces unless two adapters add leverage. Queue's backend port has production transport/realtime and test adapters; Identity's token/session port has browser-session and test adapters; Models and Nodes use production event/transport adapters; their sinks (model-load activity, node execution) are injected into Queue's runtime at the App composition root, with test doubles in Queue's runtime tests. Gallery, Generation, Workflow, and Upscale keep their compilation and UI seams internal until their feature tickets demonstrate another independent adapter. Workbench image actions likewise remain internal rather than becoming a shallow Image Feature.

| Feature    | Responsibility and representative callers                                                                                                                 | Allowed dependencies and candidate public interface                                                                                                         |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Identity   | Session/token policy and user administration; App guards/login, Queue capabilities, Workbench account/Launchpad surfaces                                  | Platform plus Models' public capability reads; session reads/commands, token port, user queries and manager contribution                                    |
| Models     | Catalog, configuration, install/load lifecycle; Generation, Queue, Workflow, Upscale, Identity, model manager                                             | Platform plus Identity's public authorization reads; descriptors/capabilities, catalog/load commands and manager contribution                               |
| Nodes      | Custom-node catalog/templates and execution lifecycle; Workflow editor and node manager                                                                   | Platform; descriptors, catalog commands, execution snapshots and manager contribution                                                                       |
| Queue      | Backend queue state, realtime interpretation, submission/reconciliation and Queue UI; App runtime, Workbench aggregate/shell, Generation/Gallery/Workflow | Platform and public envelopes/capabilities from Identity, Models, Generation, Workflow, Upscale; backend port, queue reads/commands and widget contribution |
| Gallery    | Image collection, selection, board policy and Gallery UI; App adapter, Generation, Workflow, Upscale, Workbench image actions                             | Platform and Queue's public progress reads; gallery reads/commands and widget contribution                                                                  |
| Generation | Generation settings, validation, graph compilation and Generate UI; Workbench invocation, Queue, Gallery, Workflow, Upscale                               | Platform and public Models/Gallery/Queue interfaces; settings/validation, compiled submission envelope and widget contribution                              |
| Workflow   | Workflow document, validation, compilation, preview/editor/library; App adapter, Workbench invocation, Nodes                                              | Platform and public Nodes/Models/Gallery/Queue interfaces; document reads/commands, compiled submission envelope and widget contribution                    |
| Upscale    | Upscale settings, validation, compilation and UI; Workbench invocation and Queue                                                                          | Platform and public Models/Gallery/Queue interfaces; value/validation reads, submission command and widget contribution                                     |
