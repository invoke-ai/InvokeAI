واجهة مستخدم Invoke ممكنة بفضل العديد من المساهمين والمكتبات مفتوحة المصدر. شكرًا لكم!

## بيئة التطوير

اتبع دليل [بيئة التطوير](/development/setup/dev-environment/) للإعداد. شغّل واجهة المستخدم باستخدام `pnpm dev`.

## نصوص الحزمة

- `dev`: تشغيل الواجهة الأمامية في وضع التطوير، مع تمكين إعادة التحميل السريع
- `build`: تشغيل جميع الفحوصات (dpdm, eslint, prettier, tsc, knip) ثم بناء الواجهة الأمامية
- `lint:dpdm`: فحص التبعيات الدائرية
- `lint:eslint`: فحص جودة الكود
- `lint:prettier`: فحص تنسيق الكود
- `lint:tsc`: فحص مشاكل الأنواع
- `lint:knip`: فحص الصادرات أو الكائنات غير المستخدمة
- `lint`: تشغيل جميع الفحوصات بشكل متزامن
- `fix`: تشغيل `eslint` و `prettier`، وإصلاح المشكلات القابلة للإصلاح
- `test:ui`: تشغيل `vitest` مع واجهة الويب

## توليد الأنواع

نستخدم [openapi-typescript] لتوليد الأنواع من مخطط OpenAPI للتطبيق. يتم إيداع الأنواع المُولدة في المستودع في [schema.ts].

إذا أجريت تغييرات على الواجهة الخلفية، من المهم إعادة توليد أنواع الواجهة الأمامية:

```sh
cd invokeai/frontend/web && python ../../../scripts/generate_openapi_schema.py | pnpm typegen
```

على macOS و Linux، يمكنك تشغيل `make frontend-typegen` كاختصار للمقتطف أعلاه.

## التوطين

نستخدم [i18next] للتوطين، ولكن الترجمة إلى لغات غير الإنجليزية تتم على مشروع [Weblate] الخاص بنا.

يجب تغيير سلاسل المصدر الإنجليزية فقط (أي `en.json`) في هذا المستودع.

## VSCode

### مثال على تكوين المصحح

```jsonc
{
  "version": "0.2.0",
  "configurations": [
    {
      "type": "chrome",
      "request": "launch",
      "name": "Invoke UI",
      "url": "http://localhost:5173",
      "webRoot": "${workspaceFolder}/invokeai/frontend/web"
    }
  ]
}
```

### التطوير عن بُعد

لاحظنا مشكلة انقطاع اتصال متقطعة مع إعادة توجيه المنفذ عن بُعد في VSCode.

نقترح تعطيل ميزة إعادة توجيه المنفذ في المحرر والقيام بذلك يدويًا عبر SSH:

```sh
ssh -L 9090:localhost:9090 -L 5173:localhost:5173 user@host
```

## إرشادات المساهمة

شكرًا لاهتمامك بالمساهمة في واجهة مستخدم Invoke!

يرجى اتباع هذه الإرشادات عند المساهمة.

## تحقق قبل استثمار وقتك

يرجى التحقق قبل استثمار وقتك في أي شيء غير إصلاح بسيط، في حال تعارضه مع عمل قيد التنفيذ أو عدم توافقه مع رؤية التطبيق.

إذا لم يكن طلب ميزة أو مشكلة موجودة بالفعل للشيء الذي تريد العمل عليه، يرجى إنشاء واحد.

راسل `@psychedelicious` على [discord] في قناة `#frontend-dev` أو في طلب الميزة / المشكلة التي تريد العمل عليها - يسعدنا التحدث.

## اتفاقيات الكود

- هذا تطبيق معقد إلى حد ما مع شجرة مكونات عميقة. يرجى استخدام الحفظ المؤقت (`useCallback`, `useMemo`, `memo`) بحماس.
- إذا كنت بحاجة إلى إضافة حالة عمومية عابرة، يرجى استخدام [nanostores] إن أمكن.
- كن حذرًا مع منتقي redux الخاص بك. إذا كانت بحاجة إلى أن تكون مُعلمة، ففكر في إنشائها داخل `useMemo`.
- لا تتردد في استخدام `lodash` (عبر `lodash-es`) لتوضيح نية الكود الخاص بك.
- يرجى إضافة تعليقات تصف "السبب"، وليس "الكيفية" (إلا إذا كان الأمر غامضًا حقًا).

## تنسيق الالتزام

يرجى استخدام مواصفات [conventional commits] لواجهة الويب، مع نطاق "ui":

- `chore(ui): bump deps`
- `chore(ui): lint`
- `feat(ui): add some cool new feature`
- `fix(ui): fix some bug`

## الاختبارات

لا نقوم بأي اختبار لواجهة المستخدم في هذا الوقت، ولكن فكر في إضافة اختبارات للمنطق الحساس.

نستخدم `vitest`، ويجب أن تكون الاختبارات بجانب الملف الذي تختبره. إذا كان المنطق في `something.ts`، فيجب أن تكون الاختبارات في `something.test.ts`.

في بعض الحالات، قد نرغب في اختبار الأنواع. على سبيل المثال، إذا كنت تستخدم `zod` لإنشاء مخطط يجب أن يطابق نوعًا مُولدًا، فمن الأفضل إضافة اختبار لتأكيد تطابق الأنواع. استخدم `tsafe`'s assert لهذا.

## تقديم PR

- تأكد من أن فرعك مرتب. استخدم rebase تفاعلي لتنظيف تاريخ الالتزام وإعادة صياغة رسائل الالتزام إذا لم تكن وصفية.
- شغّل `pnpm lint`. بعض المشكلات قابلة للإصلاح تلقائيًا باستخدام `pnpm fix`.
- املأ نموذج PR عند إنشاء PR.
  - لا يحتاج إلى أن يكون مفصلًا جدًا، ولكن لقطة شاشة أو فيديو جميل إذا غيرت شيئًا بصريًا.
  - إذا كان القسم غير ذي صلة، احذفه.
